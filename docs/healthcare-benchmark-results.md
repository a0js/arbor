# Healthcare Dataset: Benchmark Results & Findings

This document records the results of building and benchmarking a synthetic
healthcare test dataset against the authorization engine, and the two
performance issues it surfaced in `arbor-index-snapshot`/`arbor-authorizer`.
Generator, correctness suite, and benchmark tool live in `benches/src/bin/`;
see `docs/connectors.md` for the connector features (`parent_ids`,
`actions.csv`/`action_sets.csv`, attribute ingestion, condition parsing)
built to support this dataset.

## Dataset shape

Generated by `gen_healthcare_dataset.rs` (defaults shown; all tunable via CLI flags):

| | |
|---|---|
| Entities | 97,352 (Clinic, SpecialtyGroup, PatientRoster, Physician, Nurse, Technician, AdminStaff, Patient, RoomEquipment) |
| Policies | 4,534 |
| Multi-parent DAG | Physicians (clinic + specialty group), ~2,700 multi-clinic patients |
| Policy shapes | RBAC (`EntityType`), ReBAC (`EntityWithDescendants` over a genuine multi-parent DAG), explicit ACL (`Entity`), ActionSets, forbid-overrides-permit, ABAC (condition on patient consent attribute) |

Regenerate with:
```
cargo run -p arbor-bench --bin gen_healthcare_dataset -- --out-dir data/healthcare
```

## Correctness verification

`verify_healthcare_dataset.rs` loads the dataset through the real
connector → `Graph` → `Snapshot` pipeline and checks `AuthorizerEngine::check()`
against 10 cases covering every policy shape above (fixtures derived from the
generated CSVs directly, not re-implemented generator formulas):

```
cargo run -p arbor-bench --bin verify_healthcare_dataset
```

All 10 pass, including:
- Clinic-scoped RBAC/ReBAC
- Cross-clinic ReBAC via the physician↔specialty-group multi-parent DAG
- The ABAC consent condition gating that same cross-clinic grant (both consenting and non-consenting patients)
- Forbid-overrides-permit for break-glass restricted patients
- Narrow `ActionSet` referral grants (labs+imaging, not full chart)
- The admin billing/chart Treatment-Payment-Operations boundary

## Benchmark tool

`bench_healthcare_engine.rs` times `check()` and `list_entities()` against
real dataset entities, grouped by which kind of policy match decides each
call — not one fixed synthetic scenario:

```
cargo run -p arbor-bench --bin bench_healthcare_engine --release -- --check-iters 20000 --list-iters 200
```

Per-call timing (not `capacity.rs`'s batched-average style) so every number
here includes a constant `Instant::now()` overhead — treat absolute figures
as an upper bound; relative comparisons between cases are what matter.

## Finding #1: lopsided-array worst case in `merge_and_filter_sorted`

**Symptom:** two `check()` cases (a policy-dense patient, and a referral
grant) cost ~1,360ns vs. ~85-250ns for every other case, despite the
resource's effective-policy-set size only differing by 14 vs 17 entries.

**Root cause:** `Snapshot::merge_and_filter_sorted`'s two-pointer merge is
`O(a.len() + b.len())` by iteration count regardless of overlap — a
non-matching step only advances one pointer without doing real work. Every
physician's `effective_principal_of` is ~2,705 entries, dominated by the
~2,700 individually `Entity`-targeted break-glass forbid policies (which
apply broadly via `EntityType(Physician)` matching). A patient whose only
match is one of those entries (or a referral, similarly high-indexed) forces
the merge to scan almost the entire 2,705-entry side to find it — confirmed
by direct instrumentation (steps=2,719 vs steps=14-15 for the fast cases).

**Considered and ruled out:** reverting to the pre-arena `RoaringBitmap`
representation. Measured directly (same slices, both approaches) — the old
`RoaringBitmap` 3-way AND was no faster (1,755ns vs 1,318ns) for the
pathological case. Traced further: at 2,705 elements, Roaring stores the set
as an `ArrayStore` (`ARRAY_LIMIT = 4096` in `roaring-0.11.2`), and
`ArrayStore::bitand` is itself a scalar sorted-merge — the same algorithmic
shape as the hand-written merge, not a word-parallel bitwise AND. (Roaring's
`simd` feature isn't enabled in this workspace either, so no vectorization
was being left on the table.) The RAM-saving migration off `RoaringBitmap`
(prior commit) was not the cause of this regression; the issue is orthogonal
to representation choice.

**Fix:** `merge_and_filter_sorted` now checks the size ratio between its two
slices. Comparable sizes keep the existing two-pointer merge (already
optimal there). Beyond an 8x size ratio, it iterates the smaller slice and
binary-searches each element into the larger one — `O(min · log(max))`
instead of `O(max)`. Verified against a 240-combination property test
(random sizes/overlaps, including the exact 17-vs-2,705 shape) checked
against a naive-merge oracle.

**Result:**

| case | before | after |
|---|---|---|
| most policy-dense patient | ~1,360-1,470ns | ~240ns |
| referral narrow grant | ~1,360-1,400ns | ~235ns |
| baseline (unaffected cases) | ~85-250ns | ~220-250ns |

## Finding #2: per-candidate allocation in `list_entities`'s conditional path

**Symptom:** `list_entities` for "patients a physician can read via the
specialty-group ABAC path" cost ~3.7ms, ~17x the admin billing list despite
returning *fewer* candidates (75,471 vs 90,000) — the difference being that
the admin case is unconditional (a flat bitmap op) while the physician case
must verify an ABAC condition per candidate.

**Root cause:** not a bug — ~34ns/candidate × ~85,362 candidates needing
verification is close to the honest cost of evaluating a condition per
entity. But of that ~34ns, ~13.5ns (~40%) was `intersect_sorted` allocating
a fresh `Vec::new()` every iteration inside `verify_conditional_candidates`'s
hot loop, rather than reusing one buffer across the whole candidate set (the
way the loop's `BytecodeVM` already was).

**Fix:** added `intersect_sorted_into`/`get_effective_policies_intersected_into`
(write into a caller-owned `Vec<u32>`, `SnapshotOps` trait + both impls), and
changed `verify_conditional_candidates` to allocate one scratch buffer before
its loop instead of one per candidate.

**Result:** ~3.7ms → ~2.75ms (~25% faster) on the same case. All 157
workspace tests and all 10 correctness checks pass unchanged.

## Current numbers (post-fix)

```
check() -- work column is the resource's effective-policy-set size:
case                                                         eff_pols    avg_ns
baseline: same-clinic RBAC, unconditional                          14       223
ReBAC via multi-parent group, condition evaluates TRUE             14       246
ReBAC via multi-parent group, condition evaluates FALSE            14       240
most policy-dense patient in the dataset                           17       240
referral: narrow Entity->Entity ActionSet grant                    17       231
admin billing: EntityType->EntityType, unconditional               14        56
technician room access: EntityType->EntityType                      1        55

list_entities() -- work column is the number of entities returned:
case                                                         n_result    avg_ns
list_resources: patients a physician can read_chart (broad ReBAC + ABAC)      75471   2752041
list_resources: rooms a technician can access (narrow, unconditional)       4000      7177
list_resources: patients admin can read_billing (broad, unconditional)      90000    215948
list_principals: physicians who can read a plain patient's chart        600     33373
list_principals: physicians who can read a RESTRICTED patient's chart          0     10547
```

For context against documented V1 targets (`docs/architecture.md` /
performance notes): centralized/gRPC mode targets `list_resources()` p99 of
5-20ms at 10K resources. The broad ABAC case above returns 75K resources at
~2.75ms average — well inside that budget at ~7x the reference scale. No
further optimization pursued past this point; the remaining cost is
intrinsic to per-candidate ABAC evaluation, not an implementation defect.

## Not pursued (bigger scope, documented for later)

`resource.consent_flags.share_with_specialists == true` doesn't reference
`principal` — its truth value is the same for every physician asking. A
condition-dependency analysis at index-build time could recognize
"resource-only" conditions and precompute a bitmap of qualifying patients
once, turning per-candidate VM evaluation into a flat bitmap op at query
time. Real architectural feature, not a quick follow-up.
